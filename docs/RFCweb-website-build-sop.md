# RFCweb 官网搭建 SOP

## 0. 执行原则

本 SOP 是 RFCweb 项目后续执行的主线。后续操作必须按阶段推进，不跳步，不提前写最终页面代码。

核心原则：

- 先定需求，再定风格，再做代码，再上线。
- Google Stitch / Stage / AS Studio 的设计结果优先于临时手写页面。
- Codex 只在拿到设计稿或导出代码后，负责整理、优化、自动化和部署准备。
- 第一版只做英文官网，后续再扩展中文、西语等多语言。
- 第一版不做在线售卖，不做购物车，不做会员系统。
- 优先使用 Cloudflare Pages + GitHub，不买服务器。

## 1. 总体目标

搭建一个面向海外客户的英文公司官网，用于展示墙壁开关公司的品牌、产品、工厂实力和联系方式。

第一版目标：

- 让海外客户快速理解公司业务。
- 展示主要产品分类和代表产品。
- 建立基础信任感。
- 提供清晰的询盘入口。
- 真实上线到用户已有域名。

后续目标：

- 增加产品详情页。
- 支持产品批量自动上线。
- 扩展中文、西语等多语言版本。
- 按需增加询盘表单和数据统计。

## 2. 推荐技术方案

最终推荐：

- 网站类型：静态英文官网
- 托管平台：Cloudflare Pages
- 域名解析：Cloudflare DNS
- 代码管理：GitHub
- 产品管理：Excel / Google Sheet / CSV / JSON 表格化资料
- 产品上线方式：由 Codex 根据产品资料自动生成页面
- 服务器：第一阶段不需要

为什么这样选：

- 静态网站打开快，故障少。
- Cloudflare 对海外访问速度和稳定性更合适。
- 不买服务器可以减少月费和维护工作。
- 产品展示型官网不需要数据库和复杂后台。

## 3. 阶段一：需求与资料准备

### 你需要完成

- 确认官网第一版只做英文。
- 准备公司英文名称。
- 准备 logo。
- 准备公司英文介绍。
- 准备联系方式：邮箱、电话、WhatsApp、地址。
- 准备产品大类。
- 准备代表产品图片和资料。
- 准备工厂、展厅、生产线、证书等图片。

### Codex 负责

- 帮你整理网站信息结构。
- 帮你检查资料是否足够支撑第一版官网。
- 帮你把中文资料改写成自然的英文网站文案。
- 帮你梳理产品资料字段，方便后续自动上线。

### 进入下一阶段的条件

必须至少具备：

- 公司名称
- logo 或临时文字 logo
- 联系方式
- 至少 3 个产品分类
- 至少 3 张产品或场景图片
- 网站第一版风格方向

## 4. 阶段二：Google Stitch / Stage 设计

### 你需要完成

在 Google Stitch / Stage 里完成英文官网设计方向。

建议至少包含这些模块：

- 首页首屏
- 产品分类展示
- 公司优势
- 应用场景
- 工厂实力
- 认证资质
- 联系入口

你需要重点确认：

- 网站整体风格
- 首屏视觉感觉
- 颜色和字体方向
- 产品卡片展示方式
- 页面模块顺序

### Codex 负责

- 不提前替你重写最终页面。
- 可以帮你写给 Google Stitch / Stage 的英文提示词。
- 可以帮你评估设计是否适合海外 B2B 客户。
- 可以帮你判断页面结构是否缺少关键信任信息。

### 禁止事项

在这个阶段，Codex 不应直接进入最终前端开发。

### 进入下一阶段的条件

必须拿到以下任意一种设计交付物：

- Google Stitch / Stage 截图
- AS Studio 原型
- 导出的 HTML / CSS / JS
- Figma 或其他设计稿链接

优先交付物：

- AS Studio 导出的 HTML 代码包

## 5. 阶段三：AS Studio 原型与代码导出

### 你需要完成

- 把设计稿在 AS Studio 中整理成接近真实网站的结构。
- 确认页面跳转和模块顺序。
- 导出 HTML 代码。
- 导出相关图片、CSS、JS、字体文件。

### 交给 Codex 的资料

建议整理成一个文件夹，包含：

- `index.html`
- CSS 文件
- JS 文件
- 图片文件夹
- 字体文件夹，如有
- 产品资料表格，如有
- 设计截图，如有

### Codex 负责

- 检查导出代码结构。
- 判断哪些代码可以保留，哪些需要清理。
- 整理成可部署的网站项目。
- 不擅自改变已确认的视觉方向。

### 进入下一阶段的条件

Codex 必须已经拿到设计导出文件或明确的视觉参考。

## 6. 阶段四：Codex 代码整理与优化

### Codex 负责

拿到设计导出代码后，Codex 才开始正式改代码。

主要工作：

- 整理 HTML、CSS、JS 文件结构。
- 修复图片路径。
- 修复按钮和导航链接。
- 优化手机端排版。
- 优化图片加载速度。
- 补充基础 SEO。
- 添加 `robots.txt` 和 `sitemap.xml`。
- 准备 Cloudflare Pages 部署配置。
- 保留未来多语言扩展空间。

### 你需要确认

- 页面内容是否准确。
- 联系方式是否正确。
- 产品分类是否正确。
- 图片是否可以公开使用。
- 是否允许进入部署阶段。

### 禁止事项

Codex 不应：

- 擅自改品牌定位。
- 擅自替换你已经确认的设计风格。
- 擅自删除你提供的文件。
- 擅自购买服务或修改域名。

## 7. 阶段五：产品自动上线准备

第一版可以先做产品分类展示，不必马上做完整自动化。

后续产品数量变多后，推荐使用表格驱动方式。

### 产品资料建议字段

- product_name_en
- slug
- category_en
- series_en
- model
- main_image
- gallery_images
- colors
- material
- rated_voltage
- rated_current
- certifications
- applications_en
- description_en

### 你负责

- 提供产品表格。
- 提供产品图片。
- 告诉 Codex 哪一批产品可以上线。

### Codex 负责

- 检查表格格式。
- 批量生成产品卡片。
- 批量生成产品详情页。
- 更新分类页。
- 压缩图片。
- 重新生成 sitemap。
- 检查并部署。

## 8. 阶段六：本地预览与验收

### Codex 负责

- 启动本地预览。
- 检查电脑端。
- 检查手机端。
- 检查图片加载。
- 检查按钮链接。
- 检查 SEO 基础信息。

### 你负责

- 看预览效果。
- 确认文案。
- 确认图片。
- 确认联系方式。
- 确认是否可以上线。

### 验收标准

- 首页能正常打开。
- 手机端没有明显错位。
- 产品展示清楚。
- 联系入口可点击。
- 图片没有丢失。
- 英文内容没有明显错误。
- 页面加载速度基本正常。

## 9. 阶段七：Cloudflare Pages 上线

### 你需要完成

- 注册或登录 GitHub。
- 注册或登录 Cloudflare。
- 登录域名购买平台。
- 按 Codex 提供的步骤修改 DNS。

### Codex 负责

- 准备 GitHub 仓库所需代码。
- 准备 Cloudflare Pages 构建说明。
- 告诉你每一步点哪里、填什么。
- 检查上线后的访问状态。

### 推荐部署设置

- Build command：根据项目实际情况填写。
- Output directory：通常是 `dist` 或导出后的静态目录。
- Production branch：正式上线分支。

### 费用预估

- Cloudflare Pages：0 美元/月
- GitHub：0 美元/月
- HTTPS：0 美元/月
- 域名：用户已有，只需续费
- 服务器：不需要
